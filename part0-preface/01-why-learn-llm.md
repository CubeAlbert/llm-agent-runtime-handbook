# 为什么要学习LLM

## 时代的转折点

如果说 LLM 正在改变世界，那么这一切的起点可以追溯到一篇论文和一个思想。

2017 年 6 月，Ashish Vaswani 等八位谷歌研究者提交了 *Attention Is All You Need*[^1]。这篇论文提出了 Transformer 架构，彻底抛弃了当时主流的 RNN 等递归结构以及 CNN 等层级结构，转而使用纯粹的 Attention 机制。这一决定的工程意义极其深远：RNN 的计算必须一步一步进行，而 Transformer 可以天然并行计算，因此模型可以更充分地利用 GPU 等现代硬件进行并行计算，极大提升了训练效率和可扩展性。虽然 Attention 本身仍然存在计算和显存开销，但相比需要逐步计算的 RNN，Transformer 首次让超大规模语言模型的训练成为现实。没有这一步，后面所有的故事都无法发生。

紧接着的 2018 年 6 月，OpenAI 的 Alec Radford 发表了一篇博客——*Improving Language Understanding with Unsupervised Learning*[^2]。这篇博客提出了两个核心思想。第一，**走无监督学习路线**：不依赖昂贵的人工标注，直接在海量文本上训练语言模型，再把预训练好的模型在小规模监督数据上微调。第二，**一个模型做所有事**——"can we develop one model, train it in an unsupervised way on a large amount of data, and then fine-tune the model to achieve good performance on many different tasks?" 实验给出了肯定的回答：同一个核心模型在常识推理（COPA、ROCStories）、阅读理解（RACE）等差异极大的任务上，只需极少的适配就能达到当时的最好成绩。OpenAI 自己也在这篇博客的 "Future" 章节中预言了接下来的方向："using the well-validated approach of more compute and data"——他们从一开始就知道，这条路本质上是规模问题。

在 GPT 路线确立之后，OpenAI 又回答了另一个关键问题：模型究竟能不能一直做大？2020 年发表的《Scaling Laws for Neural Language Models》[^3] 给出了答案。研究发现，随着模型参数、训练数据和计算量不断增加，模型性能的提升并不是随机发生的，而是遵循一种可以预测的规律。更重要的是，这意味着能力的提升开始表现出一种连续积累的特征，而不是依赖某一次新的算法发明。后来人们观察到，许多此前不存在的能力，会随着模型规模增长逐渐涌现（emerge），而不是被单独设计出来。从此以后，“更多的数据、更多的算力、更大的模型”不再只是一个大胆的设想，而成为整个行业共同遵循的发展路线。

沿着这条路，OpenAI 进一步验证了这一判断。同年发布的 GPT-3 将参数规模提升到 1750 亿，并展示了另一种令人惊讶的能力：很多任务不再需要重新训练模型，只需要在 Prompt 中给出几个示例（Few-shot），甚至不给示例（Zero-shot），模型就能够完成任务。对于许多任务来说，Prompt 更像是在激活模型已经学到的能力，而不是重新教会它如何完成任务。换句话说，Prompt 并没有把知识写进模型，而是在引导模型从已经学习到的大量统计模式中，选择更合适的生成路径。

2022 年 1 月，Google 的 Jason Wei 等人发表了 *Chain-of-Thought Prompting Elicits Reasoning in Large Language Models*[^4]。这篇论文揭示了一个重要现象：**如果要求模型把思考过程一步一步写出来，而不是直接给答案，它解决复杂问题的能力会显著提升**。一个拥有 540B 参数的模型，仅凭 8 个推理示例，就在 GSM8K 数学题集上超过了经过专门微调的 GPT-3 系统。它意味着，Prompt 不仅是在告诉模型“做什么”，还可以改变模型生成文本时所遵循的推理轨迹（reasoning trajectory）。对于一个本质上不断预测下一 Token 的系统而言，改变上下文，也就改变了后续 Token 的条件概率分布，因此生成出的推理过程也会发生变化。推理开始从一个需要重新训练的目标，变成了一个可以通过 Prompt 激发出来的能力。

2025 年初，DeepSeek 推出了 R1 推理模型[^5]，成为这一方向的重要引爆点。如果说 Chain-of-Thought 证明了合理的 Prompt 可以激发模型的推理过程，那么 R1 所代表的新一代推理模型，则进一步尝试把这种推理方式通过训练内化到模型本身。模型不再依赖用户提供完整的推理示例，而是在面对数学、编程和科学等复杂任务时，能够主动生成长链条的思考过程，并利用强化学习不断优化这种能力。从 Transformer 到 GPT-3，从 Few-shot 到 Chain-of-Thought，再到 Reasoning Model，一条清晰的线索贯穿始终：LLM 表现出来的能力，正在从"学习语言模式"逐渐扩展到"学习推理模式"。至于这种能力是否真正意味着"思考"，仍然存在大量讨论，但至少在工程实践中，它已经能够完成越来越复杂的推理任务。

[^2]: Alec Radford, "Improving Language Understanding with Unsupervised Learning", OpenAI Blog, 2018.  
[^3]: Kaplan et al., "Scaling Laws for Neural Language Models", arXiv:2001.08361, 2020.  
[^4]: Wei et al., "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models", arXiv:2201.11903, 2022.

## LLM不是魔法

### 为什么看起来像魔法

LLM 能做推理、写代码、翻译——这些看起来都像人类的智能行为。但如果我们掀开盖子往里看，就会发现，它在推理阶段始终围绕着同一个目标工作：

**给定前面的所有文字，计算下一个 Token 出现的概率，然后选择一个 Token 输出。**

> 用数学符号写就是 `P(NextToken | PreviousTokens)`。  
> 需要注意的是，这里的概率并不是词典意义上的“下一个词出现概率”，而是在当前上下文条件下，对整个语言分布进行建模后的条件概率。

LLM内部当然包含复杂的神经网络结构（Transformer、Attention、MLP等），但这些结构最终共同完成的任务只有一个：**根据已有上下文估计下一Token出现的条件概率分布。**

它没有像人那样的自主意识、意图或者主动思考过程，也不会先决定“我要表达什么”，再组织语言。对于模型来说，它只是不断重复同一个过程：

> 根据当前上下文预测下一个 Token，把这个 Token 加入上下文，再预测下一个 Token，如此循环。

很多人第一次接触这里都会产生疑问：既然模型只是预测下一个Token，为什么还能表现出规划、推理甚至编程能力？

答案来自统计学习。

想象你读过世界上几乎所有公开的书籍、论文、代码仓库和对话记录。你会发现许多模式：
- 在 `1+1=` 之后，下一个词几乎总是 `2`。
- 在"光合作用的化学方程式是"之后，下一个词大概率是 `6CO₂ + 6H₂O`。
- 在"先帝创业未半而中道"之后，`崩殂` 的概率极高。

LLM 本质上就是在海量文本上学习这种统计规律。它学习的不是某一本书，而是整个文本世界中大量模式之间的关系。当你输入一个开头，它便根据学习到的概率分布预测下一个 Token，再利用更新后的上下文继续预测，最终形成一段完整的回答。

更重要的是，人类留下的大量文本中，本身就包含了推理过程、证明过程、代码设计过程以及问题求解过程。这些过程同样是一种具有统计规律的Token序列。模型在训练过程中学习到这些模式，因此生成出来的文本，也会表现出类似推理的结构。

因此，我们看到的推理、写代码、分析问题等能力，并不是模型内部存在一个“小人在思考”，而是统计学习在超大规模数据、参数和计算资源上的结果。

**这不是魔法，而是统计学习的结果。**

### 幻觉：概率的必然产物

理解了 LLM 本质上是一个预测下一 Token 的概率模型，就很容易理解它为什么会产生一个著名现象：

**幻觉（Hallucination）。**

幻觉并不是模型偶尔犯的错误，而是概率生成机制天然可能产生的结果。

**第一，统计相关不等于事实正确。** 训练数据告诉模型的是: 在某种上下文之后，哪些文本最经常出现。"A 之后经常出现 B"，不代表 A 在现实世界里必然导致 B。模型学到的是相关性，不是因果，更不是真理。当训练数据中存在错误信息、过时信息或者互相矛盾的信息时，模型会忠实地"学习"这些错误的统计模式。

**第二，模型不会先判断“自己知道不知道”。** 对于模型而言，它只是不断预测下一个 Token。“我不知道”也只是众多可能输出中的一种 Token 序列，而不是一种独立的决策机制。因此，当上下文信息不足时，它不会自动停下来，仍然需要继续生成后续 Token。如果统计上“给出一个看起来合理的答案”比输出“我不知道”概率更高，它就会继续猜测。从人类视角来看，这种猜测就形成了幻觉。

**第三，当概率分布变得平坦时，随机性会迅速增加。** 当上下文不在训练数据的覆盖范围内（即遇到陌生情况），各候选词的概率差异很小，采样结果的随机性极高——就像从一本没读过的书里随机翻一页来回答问题。

**第四，模型优化的是“最可能的文本”，而不是“最正确的事实”。** LLM在努力生成统计意义上最符合上下文的文本，而不是主动验证事实是否真实。换句话说，它优化的是语言建模目标（language modeling objective），而不是事实正确性目标（truth objective）。因此：流畅，不一定正确；合理，不一定真实；自信，不一定可靠。这也是为什么现代 AI 系统越来越多地引入检索（RAG）、工具调用（Tool Calling）、代码执行、外部数据库以及各种验证机制。模型负责生成候选答案，而系统负责验证答案。

因此，不应该把 LLM 的输出直接当作事实。它输出的是概率生成的文本，而不是精确检索得到的知识。

我们可以把它理解为：“统计意义上最可能这样写”，而要验证它"事实上是否正确"——那是另一件事，也是本书后续要讨论的核心问题。

## 从使用者到构建者

如果我们仅仅停留在使用者的视角，那么本书到这里就可以结束了。

理解 LLM 的本质是一个预测下一 Token 的概率模型，知道如何编写 Prompt，意识到幻觉的存在，对模型看似聪明的回答保持审慎而非全盘信任——对于绝大多数人来说，这已经足够了。

但是，如果我们换一个身份，从一个 AI 系统构建者的角度来看，问题才刚刚开始。

对于使用者而言，他们关心的是：

> 模型能不能回答我的问题？

而对于构建者而言，他们更关心的是：

> 当模型答错的时候，整个系统还能不能可靠运行？

这是两个完全不同的问题。前者关注模型能力（capability），后者关注系统可靠性（reliability）。一个能力很强的模型，并不天然意味着能够构建一个可靠的软件系统。

LLM 本质上是一个概率模型，而概率模型天然存在不确定性。它可能产生幻觉，可能误解用户意图，也可能在陌生场景下做出看似合理却错误的推断。

因此，一个真正可靠的 AI 系统，不能把全部正确性寄托在模型本身。

Prompt 可以改善输出，但无法消除概率性；更大的模型可以降低错误率，但无法保证永远正确。我们必须在模型外围建立一套确定性的工程体系，让系统在面对一个不确定组件时，依然能够稳定、可靠、可验证地运行。

换句话说，我们需要把一个概率模型，包装成一个可靠的软件系统。

而支撑这一目标的，不再只是 Prompt Engineering，而是更完整的工程能力：Runtime、Harness、Eval，以及后续将要介绍的一系列机制。

从这一刻开始，我们讨论的重点，将不再是如何使用 LLM，而是如何构建基于 LLM 的系统。

## 工程能力的代际跃迁

### Prompt Engineering：控制输入的艺术

早期的 LLM 工程化尝试集中在 Prompt Engineering 上。所谓 Prompt Engineering，本质是通过精心构造输入文本来控制模型输出的质量——指令设计、Few-shot 示例、思维链、角色设定，都属于这一类。它的门槛极低，不需要写代码、不需要部署系统，调几段文字就能让模型表现判若两人。直到今天，这仍然是绝大多数人唯一接触到的一层工程能力。

但它的局限也同样明显。Prompt Engineering 只能控制输入，不控制系统——模型一旦开始生成，工程师就失去了控制权。它只能影响单次交互，管不了多步骤的任务流程。

一旦你想在 Prompt 的基础上引入 Workflow 控制——比如让模型调用工具、条件分支、循环迭代——问题就出现了：模型某一步出错需要重试，谁来管？工具调用超时需要降级，谁来定义？Context 超过窗口需要记忆管理，谁来做？这些问题 Prompt 解决不了。

结论很清楚：Prompt Engineering 是起点，但远远不是终点。它是单回合的优化手段，而真实世界需要的，是多回合、有状态、可恢复的工程框架。

Prompt Engineering 本质上是在控制模型看到什么（control the input），而不是控制系统发生什么（control the execution）。

### Runtime Engineering：让系统在不确定中存活

Prompt Engineering 解决不了幻觉问题——这是模型的本质，不是输入能消除的。在一个多步任务链路中，只要 LLM 在某一步返回了错误的参数，整条链路就断掉了。再好的 Prompt 也无力回天。

这时候需要的不是更好的 Prompt，而是**一套让系统在模型出错时依然能继续运行的机制**：状态管理确保失败后能从中断点恢复，超时防止系统无限等待，重试处理偶发错误，熔断和降级防止级联失败。这些策略并不增强模型的能力——模型该错还是错——但它们保证单次错误不至于让整个系统崩溃。

但一个更根本的问题随之而来：**Runtime 解决的是执行可靠性（execution reliability），而不是决策可靠性（decision reliability）。整个流程仍然由 LLM 在控制**。 Runtime 能兜住模型的错误，却管不住模型的决策方向。假设你在处理一个数据异常问题，LLM 建议把整张表删掉、从空表重建——并且执行了——这将是一场灾难。Runtime 不会阻拦这个决策，因为对它来说，这是一次"成功的工具调用"。

到这里，问题已经变了：我们需要的不仅是让系统在出错后活下来，而是需要在模型做出灾难性决策之前阻止它。这需要一套不同于 Runtime 的新工程思想。

### Harness Engineering：把控制权收回确定性系统

针对 Runtime Engineering 的问题，有人引入了 Eval——在 LLM 决策前或执行后插入验证器，用测试捕获错误再让模型修正。但 Eval 只能过滤一部分错误，原因并不是Eval没有价值，而是任何验证机制都只能验证它能够观察到的性质。对于开放式决策问题，验证覆盖范围永远有限，而模型却可以在无限大的决策空间中寻找新的路径。

举个例子。假设有一个简单的 add(a, b) 函数，你通过 Eval 发现 add(2, 3) 的结果不是 5——很好，抓到 bug 了。于是你让 LLM 去修，它直接把函数定义改成了 return 5。测试通过了，但逻辑彻底烂掉了。Eval 只看结果对不对，不看过程对不对，而 LLM 恰好擅长制造“通过测试但完全错误”的答案。

这就是 Harness Engineering 要解决的核心问题：**把系统可靠性建立在确定性的控制流上，而不是模型的自律上。**

具体怎么做？针对高风险决策，引入 Human-in-the-loop——由人来覆盖模型最关键的不确定性。比如涉及数据删除、生产环境变更、大额资金操作时，模型只负责提出建议，最终执行必须经过人的确认。同时，系统的主控权不再交给 LLM，而是交给预定义的策略和状态机。LLM 的角色从"自主控制（LLM-controlled flow）"退回到"智能决策建议"——它出主意，确定性系统做决定。

在这里，控制权的归属发生了根本转移：模型负责提供智能（Intelligence），系统负责提供控制（Control）。当智能是不确定的，控制就必须是确定的。
